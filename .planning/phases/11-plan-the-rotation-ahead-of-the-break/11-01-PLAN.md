<objective>
ROTPLAN-01 / F1: today the coach can only REACT to the sub suggester —
there is no way to review the upcoming sub before it fires or to change
who comes on. Build the shared "plan ahead" surface (Success Criterion
#3) and prove the full override→honour loop on AFL (the only sport with
mid-period rolling subs).

This plan is the FOUNDATION both requirements share, so it deliberately
lands three reusable pieces plus the AFL consumption:
1. A pure projector adapter that seeds the Game Plan projector from
   current live game state (`projectUpcomingRotation`).
2. An editable surface = the EXISTING `GamePlanModal` extended to accept
   a seeded plan + a pin callback (no new modal — Success Criterion #3).
3. A `plannedRotation` slice in the live store (persisted via
   `partialize`, keyed by gameId) — no migration, no new event.
4. F1: AFL — open the surface on the current period from the SwapCard,
   pin the imminent sub, and have the live game honour the pinned swaps
   when the sub falls due (advisory; coach still taps "Do all").

Plan 11-02 (Wave 2) reuses pieces 1–3 unchanged to deliver F2
(build the next period's lineup, all sports) — so getting the seam
shapes right here is the point.

  keeps copy-only behaviour (new props optional).
- D-02 projectUpcomingRotation is PURE and seeds projectGamePlan;
  period[0] = current on-field reality.
- D-04 persistence = live store plannedRotation slice in partialize,
  keyed by gameId. NO migration, NO new GameEventType.

planned.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| coach edits in modal -> plannedRotation (client store) | The pinned plan is client-local (Zustand persist / localStorage), keyed by gameId. No server trust, no new network input. |
| plannedRotation -> SwapCard suggestions -> persistSwap | A pin only changes which swap the UI SUGGESTS; the actual mutation flows through the existing, RLS-guarded `persistSwap` / swap server action, and only after the coach taps "Do all". |
| stale pin -> field state | A pin authored earlier may reference a player who is now injured/loaned/elsewhere; honouring it blindly could field an invalid player. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01-A | Tampering / Integrity (stale plan) | plannedRotation honour path | mitigate | D-09 validation guard: before honouring, every pinned swap is checked against the CURRENT lineup/availability (off on-field, on swappable-bench, neither injured/loaned, period matches). Any invalid pair → the whole pin is discarded and the live suggester is used. Red-first `plannedRotationHonour.test.ts` covers the stale-pin fallback. No invalid swap can be applied. |
| T-11-01-B | Elevation / unauthorized mutation | persistSwap via honoured pin | accept | No new server action or endpoint. The honoured swap is applied only through the existing `persistSwap` path, which already enforces auth/RLS. The pin influences the SUGGESTION only; the coach still confirms. |
| T-11-01-C | Cross-game / cross-tenant bleed | plannedRotation in partialize | mitigate | The slice is keyed by `gameId` and ignored on read when `gameId !== activeGameId`; cleared when the period advances past `pinnedForPeriod` (D-10).
