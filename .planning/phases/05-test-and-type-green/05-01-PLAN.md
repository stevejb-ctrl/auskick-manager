**Idempotency requirement:** The block MUST be re-runnable. `supabase db reset` runs migrations from scratch, but using `ON CONFLICT DO NOTHING` on every INSERT keeps the block safe if anyone runs `psql -f seed.sql` outside of a fresh reset.

**Implementation outline:**

```sql
-- ─── Kotara Koalas netball seed (TEST-05) ────────────────────
-- Seed-bot profile owns the team. We INSERT into auth.users
-- directly (not via auth.admin.createUser) — fragile re: password
-- hash format, BUT the seed-bot is a NEVER-LOGIN account: no
-- email_confirmed_at flip, no encrypted_password column written
-- (NULL is OK for a service-only profile that never authenticates).
--
-- If this auth.users INSERT fails on your Supabase CLI version,
-- fall through to the standalone script — see Task 3.

do $$
declare
  v_seed_bot_id uuid := '00000000-0000-0000-0000-00000000aaaa';
  v_team_id     uuid := '5ba1eb72-ee23-4b8e-9f9c-22a12fd0fc11';
  v_player_ids  uuid[];
  v_game_ids    uuid[];
  v_player_id   uuid;
  v_game_id     uuid;
  v_started_at  timestamptz;
  i             integer;
  q             integer;
begin
  -- 1. seed-bot auth.users row + profile (idempotent).
  insert into auth.users (id, email, instance_id, aud, role, created_at, updated_at)
  values (
    v_seed_bot_id,
    'seed-bot@siren.local',
    '00000000-0000-0000-0000-000000000000',
    'authenticated',
    'authenticated',
    now(),
    now()
  )
  on conflict (id) do nothing;

  -- handle_new_user trigger creates the profile automatically;
  -- ON CONFLICT covers the re-run case.
  insert into public.profiles (id, email, full_name)
  values (v_seed_bot_id, 'seed-bot@siren.local', 'Kotara Seed Bot')
  on conflict (id) do nothing;

  -- 2. Team row + auto-membership (handle_new_team trigger).
  insert into public.teams (id, name, age_group, sport, track_scoring, created_by)
  values (v_team_id, 'Kotara Koalas', 'go', 'netball', false, v_seed_bot_id)
  on conflict (id) do nothing;

  -- 3. 9 active players, jersey 1..9, single-word names matching
  --    e2e/fixtures/factories.ts PLAYER_FIRST_NAMES convention.
  --    enforce_max_players trigger caps at 15; we're under.
  insert into public.players (team_id, full_name, jersey_number, is_active, created_by)
  values
    (v_team_id, 'Alicia',  1, true, v_seed_bot_id),
    (v_team_id, 'Brendan', 2, true, v_seed_bot_id),
    (v_team_id, 'Camille', 3, true, v_seed_bot_id),
    (v_team_id, 'Damian',  4, true, v_seed_bot_id),
    (v_team_id, 'Elena',   5, true, v_seed_bot_id),
    (v_team_id, 'Felix',   6, true, v_seed_bot_id),
    (v_team_id, 'Gemma',   7, true, v_seed_bot_id),
    (v_team_id, 'Harvey',  8, true, v_seed_bot_id),
    (v_team_id, 'Ingrid',  9, true, v_seed_bot_id)
  on conflict (team_id, jersey_number) do nothing;

  -- 4. 5 simulated games, all completed, scheduled rounds 1..5.
  --    Use deterministic UUIDs for reproducible season history.
  v_game_ids := array[
    'aaaaaaaa-0001-4a4a-aaaa-000000000001'::uuid,
    'aaaaaaaa-0002-4a4a-aaaa-000000000002'::uuid,
    'aaaaaaaa-0003-4a4a-aaaa-000000000003'::uuid,
    'aaaaaaaa-0004-4a4a-aaaa-000000000004'::uuid,
    'aaaaaaaa-0005-4a4a-aaaa-000000000005'::uuid
  ];

  for i in 1..5 loop
    insert into public.games (
      id, team_id, opponent, scheduled_at, round_number,
      status, created_by
    ) values (
      v_game_ids[i],
      v_team_id,
      'Seed Opp R' || i,
      now() - (interval '7 days' * (6 - i)),  -- rounds 1..5 backdated 5..1 weeks
      i,
      'completed',
      v_seed_bot_id
    )
    on conflict (id) do nothing;
  end loop;

  -- 5. Pull the 9 active player UUIDs in jersey order so the
  --    lineup_set events reference real player ids.
  select array_agg(id order by jersey_number) into v_player_ids
    from public.players
    where team_id = v_team_id and is_active = true;

  -- 6. For each game, seed minimal NETBALL-02-feeding event history:
  --       lineup_set (positions GS..GK = jersey 1..7, bench = 8..9)
  --       quarter_start × 4 + quarter_end × 4
  --       game_finalised
  --    All metadata.sport = 'netball' so replayNetballGame matches.
  --    Skip if game_events already has rows for this game (idempotent).
  for i in 1..5 loop
    v_game_id := v_game_ids[i];
    if exists (select 1 from public.game_events where game_id = v_game_id limit 1) then
      continue;
    end if;

    v_started_at := now() - (interval '7 days' * (6 - i));

    insert into public.game_events (game_id, type, metadata, created_by, created_at)
    values (
      v_game_id,
      'lineup_set',
      jsonb_build_object(
        'lineup', jsonb_build_object(
          'positions', jsonb_build_object(
            'gs', jsonb_build_array(v_player_ids[1]),
            'ga', jsonb_build_array(v_player_ids[2]),
            'wa', jsonb_build_array(v_player_ids[3]),
            'c',  jsonb_build_array(v_player_ids[4]),
            'wd', jsonb_build_array(v_player_ids[5]),
            'gd', jsonb_build_array(v_player_ids[6]),
            'gk', jsonb_build_array(v_player_ids[7])
          ),
          'bench', jsonb_build_array(v_player_ids[8], v_player_ids[9])
        ),
        'sport', 'netball'
      ),
      v_seed_bot_id,
      v_started_at
    );

    for q in 1..4 loop
      insert into public.game_events (game_id, type, metadata, created_by, created_at)
      values
        (v_game_id, 'quarter_start',
         jsonb_build_object('quarter', q, 'sport', 'netball'),
         v_seed_bot_id,
         v_started_at + (interval '12 minutes' * (q - 1))),
        (v_game_id, 'quarter_end',
         jsonb_build_object('quarter', q, 'elapsed_ms', 600000, 'sport', 'netball'),
         v_seed_bot_id,
         v_started_at + (interval '12 minutes' * (q - 1)) + interval '10 minutes');
    end loop;

    insert into public.game_events (game_id, type, metadata, created_by, created_at)
    values (
      v_game_id,
      'game_finalised',
      jsonb_build_object('quarter', 4, 'elapsed_ms', 600000, 'sport', 'netball'),
      v_seed_bot_id,
      v_started_at + interval '50 minutes'
    );
  end loop;

  raise notice 'Kotara Koalas seed: team=%, 9 players, 5 games', v_team_id;
end $$;
```

**Critical details:**
1. The seed-bot uses a fixed UUID (`00000000-...-aaaa`) so re-runs are stable.
2. `auth.users` minimal columns only — Supabase's full row has many more, but `id`, `email`, `instance_id`, `aud`, `role`, timestamps are the FK-required ones. NO `encrypted_password` (seed-bot never logs in). NO `email_confirmed_at` flip — irrelevant for service-only profile.
3. `handle_new_user` trigger fires AFTER INSERT on auth.users → creates profile automatically. The explicit `INSERT INTO public.profiles ON CONFLICT DO NOTHING` is a belt-and-braces safety net for older trigger versions.
4. `handle_new_team` trigger fires AFTER INSERT on public.teams → creates admin team_membership for the seed-bot. `auditKotaraKoalas` doesn't check membership; this is just a side effect of the schema's invariants.
5. NETBALL_AGE_GROUPS `'go'` config has 7 positions (gs/ga/wa/c/wd/gd/gk) and defaultOnFieldSize=7 — so 7 on-court + 2 bench = 9 players. Matches TEST-05.
6. Each game gets a quarter_start/quarter_end pair × 4 quarters → enough history for replayNetballGame to compute per-player thirds-on-court and feed NETBALL-02 tier-5 season-utilisation tiebreak.
7. The `lineup_set` metadata uses the SAME nested-Record shape that netball-quarter-break.spec.ts expects (positions: {gs:[uuid], ga:[uuid], ...}, bench: [uuid]) — verified in the spec at lines 27-28: "the replay engine's expected format".

**No source/component changes.** This is pure SQL.
