<implementation_strategy>
**Two layers of fix:**

**Layer 1 — Server-side `revalidatePath`** (netball-actions.ts):
- `endNetballQuarter` non-final branch: add `revalidatePath('/teams/${w.teamId}/games/${gameId}/live')` (and `/run/${auth.token}` for share-token path).
- `startNetballQuarter`: add the same revalidatePath calls.

This handles correctness for the next request — any subsequent server render (a navigation, a different parent-runner client connecting, a page refresh) sees the post-action data. WITHOUT this, only the in-memory client state would reflect the event.

**Layer 2 — Client-side `router.refresh()`** (NetballLiveGame.tsx + NetballQuarterBreak.tsx):
- After auto-hooter's `await endNetballQuarter(...)` resolves with `success: true`: call `router.refresh()` to trigger a server-side re-fetch of the page data. The component's useEffect that derives `quarterEnded` from game_events then sees the new state and the React tree rerenders into the Q-break branch.
- After `onStarted()` is called in NetballQuarterBreak.tsx: same pattern — `router.refresh()` so the page re-fetches and Q2's live state mounts.

`router` is from `next/navigation`'s `useRouter()`. It's NOT currently imported in either file (verified during planning — grep returned 0 hits for `useRouter` in NetballLiveGame.tsx and NetballQuarterBreak.tsx). Adds two lines per file: import + `const router = useRouter();`.

**Why both layers?**
- Server-side alone: future renders correct; current client still stale.
- Client-side alone: current client correct; but a parent-runner viewing the share link wouldn't see the change until they reloaded (their server-render would still hit cached data).

Both are small (~5 lines each), and both are the canonical Next.js pattern. CONTEXT explicitly LOCKED combining them in a single source-fix plan.

**Decision boundary on placement of router.refresh():**

Option 1 — inside the existing `startTransition` after the action:
```typescript
startTransition(async () => {
  const result = await endNetballQuarter(auth, game.id, currentQuarter, quarterLengthMs);
  if (result.success) {
    router.refresh();
  }
});
```

Option 2 — onSuccess callback chained.

**Pick Option 1** — minimal change, stays inside the existing transition boundary so the refresh fires within the same React batch as any local state cleanup.

For NetballQuarterBreak.tsx, the cleanest spot is INSIDE the existing startTransition wrapping startNetballQuarter, AFTER the `onStarted()` callback:
```typescript
const r2 = await startNetballQuarter(auth, gameId, nextQuarter);
if (!r2.success) {
  setError(r2.error);
  return;
}
onStarted();
router.refresh();
```

Alternatively, the refresh could live in NetballLiveGame.tsx's `onStarted` handler itself — but putting it adjacent to the action call keeps the cause-effect closer.
</implementation_strategy>
